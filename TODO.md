| **BUG**: heap alloc rejects a literal count | `u8 *heap = u8[256]` fails with "Expected expression" — but `u8[N]`, `u8[n]`, and even `u8[(256)]` all work. Reproduced. The feature is fine; only the spelling both docs use is broken, which is why nobody noticed: every use in the tree (`demos/bench.ap`, `demos/stress_test.ap`) happens to use an identifier count. Cause: the scanner greedily folds `type[<digits>]` into one `LangType` token with `array_size = Some(256)` (src/lexer/scanner.rs:507), then `parse_alloc` (src/parser/expressions.rs:1103) matches `TokenKind::LangType` and demands a separate `[` that no longer exists, so it backtracks and nothing accepts a bare LangType in expression position. The scanner's rewind path is exactly why identifier counts survive. `parse_type` already compensates for this same folding asymmetry (expressions.rs:905) — `parse_alloc` never got the same treatment. Fix: accept a folded `LangType` with `array_size = Some(n)` and synthesise the count. Add a literal-count regression test; the corpus has none. | HIGH |
| **BUG**: `if p { }` on a pointer | Fails with an internal codegen error — `Type error: Cannot convert value to boolean at 0:0` — for **every** pointer type, not just `u0*`. `p == null`, `p != null` and `!p` all work correctly, and the handbook lists all three as equal alternatives. Cause: `to_bool` (src/codegen/statements.rs:527) handles `is_int_value()` and `is_float_value()` and falls through to an error for pointers; the hardcoded `Position::new(0, 0)` is why the diagnostic has no source location. `!p` working proves the pointer→bool lowering exists elsewhere — this is a missing arm, not a design decision. Reproduced. | HIGH |
| Robust standard library       | **In progress, not done.** The goal: `lib/std/**` stops being a thin libc veneer. Landed so far — `std/linux/syscall` (complete, tested against the real kernel); `std/io` split into `{generic,linux,windows,libc}.ap` with all formatting on one `io_write_bytes` primitive, so the Linux path has **zero libc in its IR** (verified) and `println_i64` is 1 syscall instead of ~20; `std/mem` split with a page-level seam (`mmap` vs `VirtualAlloc`, reserve/commit explicit on both); `sqrt_f64`/`min_f64`/`max_f64`/`fabs` on SSE2. Still open: the allocator (`std/mem` is still libc `malloc` — see its own row), `std/c/string` users (`sort`/`collections`/`string`), and everything else in `lib/` that has not been looked at. **Windows has never been executed** — it compiles to IR here and nothing more; nobody has run a single Windows code path. macOS is libc-backed and equally untested. Layout convention: `std/<module>/{generic,linux,windows}.ap`, all declaring the same `$module`, gated by `$ifdef` — directory-form modules make this work with no language change, and non-transitive imports keep the platform plumbing private. | HIGH |
| `naked fn` / module-level asm | **The last language blocker for a real freestanding binary.** A libc-free binary already works today: `fn _start()` + `sys_write` + `sys_exit_group`, linked `ld -nostdlib -static`, 9,360 bytes, zero undefined symbols — verified. What does *not* work is `argc`/`argv`: at process entry they are on the stack (`argc` at `[rsp]`, `argv` at `[rsp+8]`, then envp, then auxv), not in registers, so a `_start(u32 argc, u8** argv)` reads whatever `rdi`/`rsi` happen to hold and silently sees `argc == 0` (confirmed: the binary runs, prints nothing, exits 0). Cannot be bodged with an `asm fn read_rsp()` — by the time it is called the prologue has moved `rsp` by an amount that varies with `-O` level. Needs LLVM's `naked` attribute (no prologue; body must be pure asm) or a top-level `asm { ... }` lowering to `module asm`. Deliberately kept out of `asm fn`, which passes operands *into* a function that has a prologue. | HIGH |
| Allocator on `mmap`           | `lib/std/mem/generic.ap` is still libc `malloc`/`calloc`/`free`, so any program that allocates drags libc back in — this and `std/c/string` are the only remaining libc on the Linux path. The platform seam below it already exists (`lib/std/mem/{linux,windows}.ap`: `page_reserve`/`page_commit`/`page_map`/`page_decommit`/`page_release`, mmap vs VirtualAlloc, reserve/commit exposed explicitly on both). What is missing is a real general-purpose allocator on top (free lists, coalescing, size classes). Deliberately out of scope so far: **every existing test depends on `alloc_bytes`**, so a subtly wrong allocator corrupts the whole suite silently. Do it with a test corpus of its own. | HIGH |
| Richer global initializers    | Two halves, one easy and one not. **(a) Struct literals**: `Point p = Point { x = 1, y = 2 }` is rejected today — `generate_constant_array_value` (src/codegen/globals.rs) admits only `Literal` and `FunctionRef` elements, and there is no `ConstantEmitter` path for `ExprKind::StructLiteral`. When every field is constant this is just an LLVM `ConstantStruct` and costs nothing at runtime; do this half first. **(b) Function calls**: `i32 x = compute()` cannot work as written — an LLVM global takes a *constant* initializer, full stop — so a non-constant initializer means synthesising an init function and registering it in `llvm.global_ctors`/`.init_array`. **That is run by the C runtime**, so a `-nostdlib` binary would silently never run it unless `_start` walks `.init_array` itself: decide that deliberately rather than discover it (see the freestanding rows). **Constness**: codegen currently re-derives constness by walking the initializer and erroring when it meets a non-literal. Janick's instinct is the right one — stamp `is_const_expr` on `Expression` during type checking, so codegen *asks* which path to take instead of walking-and-failing. That also buys a real diagnostic ("initializer is not a constant expression" at the offending node) instead of codegen's generic rejection. **Watch out**: global initializers are parsed in pass 1 and see only *earlier* definitions — the sole surviving exception to forward references — so `i32 x = compute()` needs `compute` declared above it, or pass 1 extended to defer initializers the way it defers bodies. | MEDIUM |

| Methods as fn-pointer values  | Possible addition: let a type function be taken as a function-pointer value, not just free functions. Today `&func` / bare `func` only names a free function (`FunctionRef` resolves via the module function table), so `demos/vm.ap` must make its opcode handlers free functions with `public` fields to fill the `(fn(VM*))[10]` dispatch table. | Direct syscalls (no libc)     | `lib/std/linux/syscall.ap`: `syscall0`..`syscall6` as `asm fn`s, the **complete** x86-64 table (all 375 numbers, generated from the kernel's own `<asm/unistd_64.h>` rather than by hand), flags, `-errno` decoding (`failed`/`errno_of` — range-test `[-4095,-1]`, not a sign test), and 35 typed wrappers. Un-wrapped syscalls go through the raw entry points, so every number is reachable. Gated `OS_LINUX` + `ARCH_X86_64`; the module is empty elsewhere, deliberately, since numbers are per-arch and Windows' move between builds. `std/io` on Linux now writes with zero libc on the path. | Function visibility + dead-code elimination | `public fn`/`public asm fn` → external linkage; everything else internal (`main`/`_start` implicitly public; `public extern`/`public alias` rejected). | Inline assembly (`asm fn`)    | A sibling of `extern fn`: where an extern fn's body lives in another object file, an `asm fn`'s body *is* its instructions. `asm fn f(i64 a: rax) -> i64: rax clobbers(rcx, memory) { "..." }` — operands pinned with `:`, `clobbers(...)` between signature and block, one string literal per asm line. Lowered to a real internal `alwaysinline` LLVM function wrapping one inline-asm call, so call sites stay ordinary calls and the existing typechecker/symbol-table/call path is untouched: a *declaration* form, not an expression form. Types and registers stay orthogonal (the type drives conversions, the register only pins location). Registers are contextual identifiers, validated against `--target` (`rax` under `--target aarch64-*` is a clean error). `sideeffect` and `~{dirflag},~{fpsr},~{flags}` are compiler-decided — omitting the flags triad silently miscompiles any caller holding a live comparison across the block. Went with the core grammar, not the metasystem hook. See doc/compiler/06-codegen.md § asm fn Lowering, doc/09 § asm fn, handbook § `asm fn`. | LOW |
